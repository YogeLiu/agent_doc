# 向量检索： ANN 那些事

向量检索是 RAG 系统里最"底层"的一个环节——上游是 embedding 模型，下游是 rerank 和生成。听起来技术含量不高，但真正做起来才发现这里坑最多：数据量大了延迟飙升、内存不够用、召回率突然下降……

这一篇聊聊向量检索的核心技术和工程实践。

---

## 精确搜索 vs 近似搜索

给定 100 万个向量，找和查询向量最相似的 Top-K，最直接的办法是暴力遍历——每个都算一遍相似度，然后排序。

这就是精确搜索（Exact NN，Nearest Neighbor）。原理简单，效果最好，但代价是 O(N × D)，N 是向量数量，D 是维度。100 万个 768 维向量，暴力算一次要 7.68 亿次浮点运算，不现实。

近似搜索（ANN，Approximate Nearest Neighbor）就是为了解决这个问题。代价是：不保证找到精确的 Top-K，可能召回率略低，但速度块几十上百倍。

ANN 的核心 trade-off 是：召回率（accuracy）和延迟的权衡。用更多的计算资源、更多的内存，能召回得更准；省资源，就召回得糙。

---

## 从低维到高维：维度灾难

理解高维空间的特性，是理解为什么需要 ANN、以及各种 ANN 算法为什么有效的关键。

### 低维空间的直觉

在 2D 或 3D 空间里，"距离"是直观的：

- 点 A 和点 B 离得近，就是真的近
- 点 A 的最近邻，就是离 A 最近的点

在低维空间里，k-means 聚类也很好用：簇的边界清晰，每个点属于哪个簇一目了然。

### 高维空间的反直觉

随着维度增加，很多"直觉"开始失效，这就是所谓的**维度灾难**（Curse of Dimensionality）。

**距离集中现象**：在高维空间里，几乎所有点之间的距离都差不多。

数学上的解释是：假设数据均匀分布在单位立方体 [0,1]^d 中，当维度 d 很大时，点到原点的距离会集中在某个狭窄的范围内。

```
以 L2 距离为例：
- 低维（d=2）：最远点和最近点的距离差异明显
- 高维（d=768）：几乎所有点到原点的距离都差不多——"远"不到哪去，"近"也近不到哪去
```

这就导致一个尴尬的问题：在高维空间里，"最近邻"这个概念变得模糊。排名第一的结果和排名第一百的结果，距离上可能没多大差别。

**聚类失效**：k-means 在高维空间里效果变差，因为：

- 簇的边界变得模糊
- 很多点其实处于"中间地带"，不属于任何一个明显的簇
- 簇的中心点可能和数据点没什么关系

### 对向量检索的影响

理解了高维空间的特性，就理解了为什么 ANN 是必须的：

1. **暴力搜索不现实**：100 万个 768 维向量，算一次相似度要 7.68 亿次运算——延迟太高

2. **传统索引失效**：KD-Tree 在低维很好用，但高维退化严重（本质上也是维度灾难的体现）

3. **近似是必然选择**：既然"精确"和"快"只能二选一，那就接受近似，用 ANN 换取可接受的召回率

这也解释了为什么现代 ANN 算法（HNSW、IVF-PQ）都是"概率性"的——它们不追求精确找到最近邻，而是在"召回率"和"延迟"之间找平衡。

## ANN 的几种主流路线

ANN 算法大体分四类：基于树、基于聚类、基于图、基于哈希。每种都有适用场景。

### 1. 基于聚类：IVF

IVF（Inverted File Index）是最经典的思路——先把向量聚成若干个簇，查询的时候先找到最可能相关的几个簇，然后只在这些簇里搜。

原理：

```
1. 训练阶段：用 k-means 把 N 个向量聚成 nlist 个簇（每个簇有个中心点）
2. 索引阶段：每个向量分配到最近的中心点对应的"倒排表"里
3. 查询阶段：
   - 先算查询向量和 nlist 个中心点的距离
   - 选出最近的 nprobe 个簇
   - 只在这 nprobe 个簇的倒排表里暴力搜索
```

IVF 把搜索范围从 N 缩小到了 nprobe × (N/nlist)。比如 nlist = 1024，nprobe = 16，那就是 16 × (100万/1024) ≈ 1.5 万个向量——比 100 万小两个数量级。

**参数怎么调**：

- nlist（聚类数量）：通常取 4√N 到 16√N。100 万向量大约在 4000-16000 之间。
- nprobe（查询时探几个簇）：这是召回率和延迟的主旋钮。调大 nprobe，召回率上升，延迟也上升。

IVF 的问题是：聚类质量依赖于数据分布。如果数据分布不均匀，某些簇特别大，查询时还是会被拖慢。更重要的是，IVF 只能和量化（PQ）配合使用才能真正省内存——否则每个簇里还是存着原始向量。

### 2. 基于图：HNSW

HNSW（Hierarchical Navigable Small World）是现在最流行的 ANN 算法。如果 IVF 是"先分桶再搜"，HNSW 就是"画张图让你导航"。

理解 HNSW 需要先了解它的前身 NSW（Navigable Small World）。

#### NSW：基础版小世界图

NSW 的核心思想：将向量组织成一个"小世界网络"——在这个图里，任意两个节点之间的路径很短。

```
构建过程：
1. 依次插入每个向量作为图的节点
2. 每个新节点会和图中已有的 M 个"最近邻"建立边
3. 早期插入的节点，因为可供选择的邻居少，往往会被选中多次，逐渐成为"枢纽节点"
```

这个设计有个问题：枢纽节点连接了太多其他节点，导致搜索时容易陷入局部最优——每次都先访问枢纽节点，但枢纽节点的邻居不一定和查询向量最相似。

#### HNSW：分层的改进

HNSW 在 NSW 基础上引入了"分层"设计：

```
构建阶段：
1. 为每个向量随机分配一个"层数" L（遵循指数分布，层数越高节点越少）
2. 层数越高的节点，只和同层的少数节点连接
3. 层数越低的节点，和更多同层及下一层的节点连接

搜索阶段（分层贪心搜索）：
1. 从最高层开始，找当前层的最近邻
2. 以这个最近邻为入口，进入下一层
3. 重复直到最底层
4. 在最底层做完整的贪心搜索，返回 Top-K
```

这个设计的精妙之处在于：

- **上层稀疏**：层数越高，节点越少，连接越稀疏。这让搜索能快速"跳"到目标区域的大致位置。
- **下层密集**：最底层节点多、连接密，确保搜索能精细地找到最近的点。
- **跳跃式搜索**：上层帮你做"粗定位"，下层做"精搜索"。平均复杂度从 O(N) 降到 O(log N)。

#### HNSW 构建过程详解

```python
# 伪代码：HNSW 插入一个新节点
def insert_node(new_vector):
    # 1. 决定这个节点的层数（指数分布）
    L = floor(-log(random()) * ml)  # ml 通常是 1 / ln(M)

    # 2. 从最高层开始，逐层向下搜索
    # 每层找到当前层的最近邻，作为下一层的入口
    entry = find_nearest_in_layer(L, new_vector)

    # 3. 从第 L 层到第 0 层，逐层插入
    for layer in range(L, -1, -1):
        # 在这一层找到 M 个最近邻
        neighbors = search_nearest_neighbors(new_vector, layer, ef=efConstruction)

        # 和这些邻居建立双向边
        for neighbor in neighbors:
            add_edge(new_vector, neighbor)
            add_edge(neighbor, new_vector)

        # 如果这一层的边数超过 M，需要裁剪
        # 保留"方向多样化"的边，删除"方向相近"的边
        if len(neighbors) > M:
            neighbors = prune_edges(neighbors, M)

        # 更新 entry，为下一层做准备
        entry = neighbors[0]  # 最近的那个
```

**层数分配的数学原理**：为什么用指数分布？

如果层数服从指数分布，那么第 0 层的节点数大约是 N/M，第 1 层是 N/M²，第 L 层是 N/M^L。这个分布保证了：层数越低，节点越密；层数越高，节点越稀疏。搜索时从最高层开始，能以最少步数跳到目标区域。

#### HNSW 搜索过程详解

```python
# 伪代码：HNSW 搜索 Top-K
def search(query_vector, K, ef):
    # 1. 从最高层开始
    current_layer = max_layer
    current_node = entry_points[current_layer]  # 最高层的入口点

    # 2. 逐层向下搜索
    while current_layer > 0:
        # 在当前层做贪心搜索：遍历邻居，找更近的
        neighbors = get_neighbors(current_node, current_layer)
        best = find_nearest(neighbors, query_vector)

        # 如果没有更近的了，就进入下一层
        if distance(query_vector, best) < distance(query_vector, current_node):
            current_node = best
        else:
            current_layer -= 1

    # 3. 在最底层做完整搜索（efSearch 控制候选数）
    # 维护一个大小为 ef 的优先队列（最小堆，按距离排序）
    candidates = [current_node]
    visited = set([current_node])
    results = []

    while candidates:
        # 取出距离查询向量最近的候选
        current = pop_nearest(candidates, query_vector)

        # 如果结果集已满，且当前候选比结果集中最差的还远，停止
        if len(results) >= ef and distance(query_vector, current) > distance(query_vector, results[-1]):
            break

        # 遍历当前节点的邻居
        for neighbor in get_neighbors(current, layer=0):
            if neighbor not in visited:
                visited.add(neighbor)

                # 如果结果集未满，或这个邻居比最差的更好，加入候选
                if len(results) < ef or distance(query_vector, neighbor) < distance(query_vector, results[-1]):
                    candidates.append(neighbor)

        # 加入结果集（保持有序）
        insert_into_results(results, current, query_vector)

    return results[:K]
```

#### 参数详解与调优

**M（每个节点的边数）**

- M 越大：图越密集，两个任意节点之间的路径更短，召回越高
- M 越大：内存占用越大，构建时间越长
- 经验值：M = 16 是通用场景的最佳起点；M = 32 适合对召回率要求极高的场景

```
内存估算：每个节点需要存 M 个邻居的指针（4 字节 × M）+ 邻居距离（可选，4 字节 × M）
100 万节点 × M=16 ≈ 100 万 × 128 字节 ≈ 128 MB（边信息）
```

**efConstruction（建图时的候选数）**

- efConstruction 越大：构建图的质量越高，搜索时的"视野"越广
- efConstruction 越大：构建时间越长（O(N × efConstruction × M)）
- 经验值：efConstruction = 200-400 是常见选择

**efSearch（搜索时的候选数）**

- efSearch 越大：召回率越高（能探索更多候选）
- efSearch 越大：延迟越高（要遍历更多节点）
- 经验值：从 efSearch = 50 开始，看召回率和延迟的平衡点

**ml（层数分布参数）**

- ml 控制层数的指数分布参数，通常 ml ≈ 1 / ln(M)
- M = 16 时，ml ≈ 0.2
- 大多数实现会限制最大层数 L_max（通常是 3-7）

#### 召回率 vs 延迟调优曲线

典型的 HNSW 调优曲线：

```
efSearch=10:  Recall@10 ≈ 85%, 延迟 ≈ 5ms
efSearch=20:  Recall@10 ≈ 92%, 延迟 ≈ 8ms
efSearch=50:  Recall@10 ≈ 96%, 延迟 ≈ 15ms
efSearch=100: Recall@10 ≈ 98%, 延迟 ≈ 30ms
efSearch=200: Recall@10 ≈ 99%, 延迟 ≈ 60ms
```

通常 efSearch = 50-100 是一个甜点区间——召回率已经很高（>96%），延迟也能接受。

#### 内存估算

100 万个 768 维向量：

| 组件       | 计算方式            | 大小                            |
| ---------- | ------------------- | ------------------------------- |
| 原始向量   | N × d × 4 bytes     | 100万 × 768 × 4 ≈ **30.5 GB**   |
| 图边信息   | N × M × 2 × 4 bytes | 100万 × 16 × 2 × 4 ≈ **122 MB** |
| 节点元数据 | N × 8 bytes         | ~8 MB                           |
| **合计**   |                     | **≈ 31 GB**                     |

如果用 int8 量化，内存可以降到 ~8 GB。

#### HNSW 为什么这么快

- **不是遍历，是导航**：图结构保证了"近邻之间有边"，搜索时能跳过大部分无关的点
- **内存局部性**：相邻的向量在内存里也相邻，CPU 缓存命中率高
- **多层跳表**：上层帮你粗定位，下层精细搜索，平均 O(log N)

### 3. 基于量化：PQ

PQ（Product Quantization，乘积量化）是另一条路——不存精确的向量了，存"压缩后的编码"。

#### 核心思想

高维向量空间有一个特性：维度越高，数据分布越稀疏。在这样的空间里，如果直接用 k-means 量化成一个 K 类别的码本，为了让每个向量都能被精确近似，K 必须非常大——可能需要几万甚至几十万才能保证量化误差足够小。这带来的问题是：训练一个这么大的码本需要海量样本，计算成本也极高。

PQ 的聪明之处是"分而治之"：不直接量化整个向量，而是把向量切成 m 个子空间，对每个子空间独立做小规模的量化。每个子空间只需要少量 centroid（比如 256 个），m 个子空间组合起来，总的表示能力依然很强。

#### 具体流程

假设一个向量是 768 维：

1. **向量子空间划分**：把向量均匀切成 m 段。比如 m = 8，每段 96 维。

2. **独立量化**：对每个子空间单独做 k-means。比如每段用 k = 256 个 centroid（8 bits），这意味着每段只需要 1 byte 来存储编码。

3. **编码存储**：一个 768 维向量，用 PQ 后变成 m × 1 byte = 8 byte。压缩了 768 × 4 / 8 = 384 倍。

#### 距离计算：查表法

PQ 查询的核心优化是"查表"，避免对每个候选向量重新计算距离。

```
假设：
- query 向量 q = [q1, q2, ..., qm]（m 段，每段是原始维度的向量）
- 码本 C = {c1, c2, ..., ck}（每个子空间有 k 个 centroid）

步骤：
1. 对每个子空间 j，计算 query 子向量 q(j) 和该子空间所有 k 个 centroid 的距离
   → 得到一个距离表 T[j][i] = dist(q(j), centroid_i)

2. 对库中的 PQ 编码（比如 [3, 7, 1, 0, 5, 2, 6, 4]），只需要：
   → 近似距离 = T[0][3] + T[1][7] + T[2][1] + ... + T[7][4]
   → 8 次查表 + 7 次加法
```

这比直接计算原始向量的距离（768 次乘法 + 767 次加法）快得多。而且查表操作完全向量化，CPU 缓存友好。

#### ADC vs SDC

- **ADC（Asymmetric Distance Computation）**：query 不量化，用原始向量；库向量量化。这是 PQ 的标准做法，精度更高。
- **SDC（Symmetric Distance Computation）**：query 和库向量都量化。两边都近似，误差叠加，精度更差，但查询时可以进一步节省计算。

工程中几乎都用 ADC。

#### 量化误差的来源

PQ 的近似误差来自两方面：

1. **子空间量化误差**：每个子空间的向量被强制归类到最近的 centroid，这个归类误差是主要来源
2. **子空间划分假设**：假设各子空间独立，但实际中高维向量的维度之间往往有相关性

#### 参数选择

| 参数 | 取值 | 说明 |
|------|------|------|
| m（子空间数） | d/8 ~ d/4 | 768 维通常用 8-12 段；m 越大，每段维度越低，量化误差越大 |
| k（码本大小） | 256 (2^8) | 通常用 8 bits，即 k = 256；实践中 k=256 是最优选择 |
| 压缩比 | d × 4 / m | 768 维 / 8 = 384 倍；768 维 / 16 = 192 倍 |

#### 召回率参考

| 压缩比 | 子空间数 m | 每段维度 | Recall@10 |
| ------ | --------- | -------- | --------- |
| 16×   | 48        | 16       | ~95%     |
| 32×   | 24        | 32       | ~90%     |
| 64×   | 12        | 64       | ~80%     |
| 128×  | 6         | 128      | ~70%     |

压缩比越高，召回率下降越明显。32 倍压缩是工程中的常见选择。

#### 对残差的量化

在 IVF-PQ 中，通常不是直接对向量做 PQ，而是对"残差"做 PQ。

```
1. 先用 IVF 聚类，得到 nlist 个 centroid
2. 每个向量减去它所属簇的 centroid，得到残差向量
3. 对残差向量做 PQ
```

为什么对残差量化更好？因为残差通常比原始向量更"集中"、方差更小，同样的码本能更精确地近似。

#### OPQ：优化子空间划分

PQ 假设各子空间独立，但原始向量的维度之间往往有相关性（比如相邻维度可能高度相关）。OPQ（Optimized PQ）的做法是：先对向量做一个正交变换（旋转），让变换后的向量在各子空间的能量分布更均匀，然后再做 PQ。

OPQ 通常能比普通 PQ 提升 1-3% 的召回率。代价是需要额外的训练过程。

### 4. IVF + PQ：大规模标配

IVF 和 PQ 可以组合在一起用——IVF 负责分桶，PQ 负责存桶里的向量。

这就是 IVF-PQ：

```
1. 先用 IVF 把向量分到 nlist 个桶
2. 对每个桶里的向量，用 PQ 编码压缩
3. 查询时：
   - 先找到最近的 nprobe 个桶
   - 在这些桶里用 PQ 近似距离搜索
   - 如果需要，可以对 Top-K 用原始向量 rerank
```

IVF-PQ 是大规模（百万到亿级）的性价比最优解——比纯 HNSW 省内存，比纯 IVF 快，比纯 PQ 召回准。

### 4.5 HNSW + PQ：另一种组合思路

除了 IVF+PQ，HNSW 和 PQ 也可以组合使用，这种组合在某些特定场景下有其价值。

**为什么需要 HNSW + PQ**

HNSW 的问题是内存占用大——100 万个 768 维向量要 ~30GB 内存，很多机器根本扛不住。PQ 可以把向量压缩存储，但纯 PQ 的问题是搜索效率不高。

HNSW + PQ 的思路是：用 HNSW 的图结构做搜索导航（快），但用 PQ 编码存储向量（省内存）。

**查询流程**

```
1. 在 HNSW 图上做搜索，找到候选节点
2. 用 PQ 编码计算候选节点的近似距离
3. 如果需要，对 Top-K 用原始向量做精确 rerank
```

**生产环境使用情况**

HNSW + PQ 组合在生产环境中的使用相对较少，原因：

1. **实现复杂度高**：需要同时维护图结构和量化编码，Qdrant 等少数向量数据库支持
2. **召回率不如 IVF-PQ**：HNSW 的搜索路径依赖精确距离，PQ 的近似会累积误差
3. **调优困难**：两套参数体系耦合，调参复杂度高

**适用场景**

HNSW + PQ 适合什么场景：
- 数据量在百万级，介于 HNSW（需要大内存）和 IVF-PQ（需要更大规模）之间
- 对延迟敏感，同时内存预算有限
- 已经用了 HNSW，想通过 PQ 降低内存

**主流向量数据库的支持情况**

| 数据库 | HNSW | IVF-PQ | HNSW+PQ |
|--------|------|--------|---------|
| Qdrant | ✓ | ✓ | ✓ (HNSW + SQ) |
| Weaviate | ✓ | ✓ | ✗ |
| Milvus | ✓ | ✓ | ✓ |
| pgvector | ✓ | ✓ | ✗ |

Qdrant 支持的是 HNSW + SQ（标量量化），而非完整的 PQ。这是一个工程折中——SQ 比 PQ 实现简单，效果差距不大。

---

### 5. 其他：LSH、KD-Tree

这两种在现代 ANN 里用得少了。

- KD-Tree：在低维（< 20维）效果好，高维退化严重。embedding 都是几百上千维，不适用。
- LSH（Locality Sensitive Hashing）：原理是用随机投影把向量映射到 hash 值，"相似的向量大概率落在同一个 bucket"。精度通常不如 HNSW 和 IVF-PQ，现在主要用在"粗筛 + 精排"的两阶段流程里。

---

## 工程选型：数据量决定一切

选什么索引类型，看数据量：

| 数据规模         | 推荐索引                 | 理由                                  |
| ---------------- | ------------------------ | ------------------------------------- |
| < 10 万          | Flat（暴力）             | 数据量小，暴力搜反而最快，延迟 < 10ms |
| 10 万 - 100 万   | HNSW                     | 延迟敏感，内存够用                    |
| 100 万 - 1000 万 | IVF-PQ 或 HNSW（量化后） | 内存开始吃紧，用量化压缩              |
| 1000 万 +        | IVF-PQ + 分片            | 必须分片了，单机存不下                |

这里的核心矛盾是：HNSW 召回准、延迟低，但内存需求大；IVF-PQ 省内存，但召回略差、延迟略高。

一个实用的策略是：先用 HNSW + int8 量化，内存能hold住就先跑起来；等数据量起来了，再切到 IVF-PQ。

## 几个常见的坑

### 1. 过滤（Filtering）和向量搜索的矛盾

很多业务需要在向量检索的同时加过滤条件——比如"只搜这个租户的文档"、"只搜最近一年的文档"。

问题在于：ANN 索引按"向量相似"组织，过滤条件按"业务属性"组织，两者不兼容。

#### 常见解法

**Post-filter（后置过滤）**

```
流程：向量检索 Top-K → 用过滤条件筛 → 返回最终结果
```

优点：实现简单，不需要改索引结构
缺点：
- 筛完后可能一个都不剩
- 召回的 K 个可能全被过滤掉，导致空结果
- 过滤比例越高，空结果概率越大

适用场景：过滤条件宽松（过滤掉 10-20% 以下）

**Pre-filter（前置过滤）**

```
流程：满足过滤条件的向量 → 在这部分做向量检索 → 返回结果
```

优点：不会有"筛掉"的问题
缺点：
- 过滤后的集合可能很小，召回率暴跌
- 需要维护一个"满足过滤条件的向量 ID 列表"，实现复杂

适用场景：过滤条件非常宽松，或者有高效的 bitset 索引

**分索引（按过滤维度建多个索引）**

```
流程：按 tenant_id 分别建索引 → 查询时只在对应索引搜
```

优点：最干净，过滤和检索完全解耦
缺点：
- 过滤维度多了就爆炸（比如同时按 tenant + 时间 + 类型建索引）
- 维护成本高

适用场景：过滤维度是强隔离的（比如多租户场景）

#### 进阶方案

**方案一：Bitset 过滤**

向量数据库内部维护一个 bitset，记录每个向量是否满足过滤条件。查询时：

```
1. 先用 bitset 过滤，排除不满足条件的向量
2. 在剩余向量上做 ANN 搜索
```

这是 Qdrant、Milvus 等主流向量数据库支持的方式。优点是实现相对简单，缺点是过滤效率依赖于数据分布。

**方案二：融合查询（Fusion Query）**

有些数据库支持"先分别查、再融合"的模式：

```
1. 向量检索返回 Top-100
2. 过滤条件匹配返回 50 个向量
3. 用 RRF 或加权融合两路结果
```

这种方案的好处是"两边都不完全依赖"，某一边出问题另一边还能兜底。

**方案三：近似过滤（Approximate Filter）**

如果过滤条件是"最近一年"这种时间相关的，可以用 ANN 索引的"范围搜索"能力：

- Milvus 支持时间字段的范围索引
- 先按时间范围粗筛，再用向量相似度精排

#### 实践建议

| 场景 | 推荐方案 |
|------|---------|
| 多租户（每个 tenant 独立数据） | 分索引 + 分库 |
| 时间范围过滤 | 范围索引 + Post-filter |
| 标签过滤（10 个以内） | 数据库内置 bitset |
| 动态多条件组合 | 应用层二次过滤 + 扩大召回 K |

一个关键原则：尽量在业务层解决过滤问题。如果"每个用户只能搜自己的数据"是强需求，最干净的做法不是在检索层过滤，而是在存储层就分开——每个 tenant 独立的库、独立的索引。

#### 过滤率对召回率的影响

| 过滤比例 | 预期影响 | 建议 |
|----------|----------|------|
| <10% | 影响小，Post-filter 即可 | 直接用向量数据库的过滤功能 |
| 10-50% | Post-filter 可能导致空结果 | 考虑 Pre-filter 或增大召回 K |
| >50% | 几乎必然丢失召回 | 考虑分索引或业务层分离 |

#### 主流向量数据库的过滤实现

**Qdrant：Filterable HNSW**

Qdrant 在 HNSW 基础上实现了"过滤感知"的图索引。核心思路是：**在构建索引时就考虑过滤条件，提前建立"过滤后的子图"**。

问题本质：HNSW 图的搜索依赖"短边+长边"的组合来探索空间。如果用了过滤条件，大量节点被排除，图的结构会被破坏——搜索路径可能断裂，导致漏掉相关结果。

Qdrant 的解决方案：

```
1. 构建阶段：为每个 payload 值创建子图
   - 比如 "brand=Apple" 的所有向量，形成一个连通的子图
   - 子图内部保持完整的 HNSW 连接

2. 查询阶段：Query Planner 自适应选择策略
   - 过滤匹配多 → 用 HNSW 搜索，但在遍历时跳过不满足条件的节点
   - 过滤匹配少 → 直接退化为全表扫描（比跳图更快）
```

关键配置：**Payload Index**

Qdrant 不会默认对 payload 建立索引，需要显式创建：

```python
# 索引 keyword 字段
client.create_payload_index(
    collection_name="products",
    field_name="brand",
    field_schema=models.PayloadSchemaType.KEYWORD,
)

# 索引数值范围字段
client.create_payload_index(
    collection_name="products",
    field_name="price",
    field_schema=models.PayloadSchemaType.FLOAT,
)

# 查询时使用过滤
results = client.query_points(
    collection_name="products",
    query=query_vector,
    query_filter=models.Filter(
        must=[
            models.FieldCondition(key="brand", match=models.MatchValue(value="Apple")),
            models.FieldCondition(key="price", range=models.Range(lte=1000)),
        ]
    ),
    limit=10,
)
```

**重要**：Payload index 最好在数据写入之前创建。如果在 HNSW 构建之后才建索引，过滤效果会打折扣，需要重建索引。

**Milvus：多层次过滤支持**

Milvus 在 2.4+ 版本增强了 metadata 过滤能力：

1. **Int 和 Float 字段的范围过滤**：
   ```python
   # Milvus 表达式过滤
   expression="price >= 500 and price <= 2000"
   ```

2. **String 字段的全文匹配**（2.4.3+）：
   - Prefix：`title like "The%"`
   - Infix：`title like "%the%"`
   - Postfix：`title like "%Rye"`
   - Wildcard：`title like "Flip_ed"`

3. **JSON 字段过滤**：Milvus 支持 JSON 类型的 payload，可以对嵌套字段过滤

4. **过滤策略**：
   - Milvus 使用"过滤后搜索"策略（类似 Pre-filter）
   - 对于强过滤场景，会先根据过滤条件筛选出候选集，再在候选集内做向量搜索

```python
# Milvus 过滤查询
from pymilvus import Collection, AnnSearchRequest

# 方式1：表达式过滤
results = collection.search(
    data=[query_vector],
    anns_field="embedding",
    expr="category == 'laptop' and price <= 2000",
    limit=10
)

# 方式2：JSON 字段过滤
results = collection.search(
    data=[query_vector],
    anns_field="embedding",
    expr="properties.color == 'red'",
    limit=10
)
```

#### 数据库选择建议

| 需求场景 | 推荐数据库 |
|----------|----------|
| 强过滤（每个用户只能搜自己数据） | Qdrant（Filterable HNSW） + 分 collection |
| 多条件动态组合过滤 | Milvus（表达式能力强） |
| 过滤 + 延迟敏感 | Qdrant（Query Planner 自适应） |
| 已有 PG 业务，想加向量检索 | pgvector（SQL 过滤） |

---

### 2. 动态更新

向量数据库经常需要增删改。

- **HNSW 删除节点**：图结构会被破坏。常用"软删除"（标记为 deleted，查询时跳过），定期重建。
- **IVF-PQ 追加向量**：可以做，新增的向量分配到最近的簇。但 PQ 码本是训练好的，新向量可能导致分布偏移，需要定期重建。
- **Flat 追加**：最简单，append-only，定期重建索引。

如果你的数据更新频繁，选索引时要考虑"更新友好度"。

### 3. 召回率突然下降

可能的原因：

- **数据分布变了**：新上的文档和之前的分布差异大，索引质量下降。解法：定期重建。
- **参数被人改了**：比如 efSearch 从 100 改到了 20。解法：review 配置变更。
- **向量版本不兼容**：同一个系统里跑了两个版本的 embedding 模型，向量分布不一致。解法：固定模型版本，定期重做索引。

## 聊聊 DiskANN

微软的 DiskANN 是近两年的一个技术突破——把索引做到磁盘上，内存压力从 TB 级降到 GB 级。

原理：

```
1. 所有向量用 PQ 压缩，存内存（很快能遍历）
2. 完整的图结构和原始向量存磁盘
3. 查询时：
   - 在内存的 PQ 编码上做近似搜索，确定候选
   - 从磁盘批量读取候选的原始向量
   - 精确重排
```

这样可以用 32GB 内存跑 10 亿级向量的检索，召回率还能保持在 95%+。Azure AI Search、Milvus 都已经支持 DiskANN。

如果你的数据量到了亿级，可以考虑这个方案。

## 小结

向量检索是 RAG 链路上最"底层"的一环，但坑一点不少。

核心结论：

1. **小数据量（< 10万）用 Flat**，暴力搜索最快最准
2. **中等规模（10万 - 100万）用 HNSW**，召回和延迟平衡最好
3. **大规模（百万以上）用 IVF-PQ**，或者 DiskANN
4. **过滤是系统工程问题**，没有完美解，业务层面解决往往比技术层面更干净
5. **更新频繁的索引要定期重建**，别忘了

很多团队的问题不是"算法选错了"，而是"参数没调对"。HNSW 的 efSearch 设成 10 和设成 100，召回率能差 10 个点。先把现有方案调明白，再考虑换算法。

## 常见问题

**Q：HNSW 和 IVF-PQ 选哪个？**

A：先看内存够不够。够，就用 HNSW + int8 量化，召回和延迟都更好。不够，就用 IVF-PQ。等数据量过了千万，再考虑 DiskANN 或分片。

**Q：PQ 的压缩比设多少合适？**

A：32 倍是甜点。Recall@10 通常还能维持在 85-90%。压到 64 倍以上，召回会掉得比较快，需要靠增大 nprobe 补救。

**Q：向量需要定期重建索引吗？**

A：需要。两种情况：一是数据分布变了（新增了大量新文档），二是删除比例超过 20-30%（大量 tombstone 会拖慢搜索）。通常每季度或每半年重建一次是比较合理的节奏。

**Q：延迟和召回怎么权衡？**

A：看业务。如果是对话式 RAG（用户等回答），延迟优先，P95 延迟不要超过 1 秒。如果是离线批处理，召回优先。线上可以用 A/B 测试看用户实际感受——延迟加了 200ms，召回率加了 5%，用户留存有没有变化？
