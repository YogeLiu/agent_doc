# 向量模型：语义编码这一步做错了，后面全错

很多人选向量模型的方式是这样的：打开排行榜，看哪个得分高，ok，就用这个。或者说别人用什么我就用什么。这种方式不能说完全错，但很容易踩坑——排行榜上的高分模型，放在你的业务场景里可能表现一团糟。

这一篇聊聊向量模型到底在做什么，以及怎么选一个真正适合你的。

---

## 向量模型到底在做什么

把一段文本变成一个向量，这个过程叫 encoding。好的 encoding 能让语义相近的内容在向量空间里离得近，语义差得多的内容离得远。

理解这个过程有个经典的例子：

```
vector('国王') - vector('男人') + vector('女人') ≈ vector('女王')
```

这个等式能成立，是因为模型在训练过程中学到了一个"语义空间"——在这个空间里，性别、皇室地位这些维度被编码成了向量方向。所以做向量减法能做语义推理。

但这个能力不是天然的。模型需要通过大规模语料的训练才能获得。训练方式主要两种：

**对比学习（Contrastive Learning）**：这是现在的主流方式。给模型看"相似的一对"和"不相似的一对"，让它学会把相似的推近、不相似的推远。SimCSE、BGE、Cohere 的训练都是这个思路。

**预测学习（Predictive Learning）**：BERT 时代的方式——Masked Language Model，让模型预测被 Mask 掉的词。这种方式训出来的向量更注重"重建"，而不是"匹配"。

这两种方式训出来的向量在检索任务上表现差异明显。对比学习训出来的向量，在相似度匹配任务上通常比 BERT-style 的好很多。所以现在做检索，基本都选对比学习路线的模型。

## 几个关键概念

### 维度（Dimension）

向量有多少维。常见的有 384、768、1024、1536。

维度越高，通常表达能力越强，但代价也越大——存储空间、计算延迟都跟着涨。

一个经验法则：在效果够用的前提下，越低越好。OpenAI 的 ada-002 默认 1536 维，很多开源模型 768 维就足够好了。

### 归一化（Normalization）

向量生成后要不要做归一化？

归一化的好处是：向量变成了单位长度，这时候"方向"成为唯一的信息来源。不管向量原来多长，归一化后都变成一样的。

**关键点**：归一化后，L2 距离和 cosine similarity 在排序上完全等价。所以如果你不确定用什么度量，归一化 + L2 是最稳妥的兜底方案。

但有些模型训练时本身就没做归一化，输出的向量长度有业务含义（比如推荐系统里"长度"代表"置信度"）。这时候别乱归一化。

### 最大输入长度

每个模型能处理的 token 数是有限的。超过这个长度，文本会被截断。

常见模型的输入限制：

| 模型                         | 最大输入 token |
| ---------------------------- | -------------- |
| text-embedding-ada-002       | 8192           |
| text-embedding-3-small/large | 8192           |
| bge-large-zh                 | 512            |
| bge-m3                       | 8192           |
| Cohere embed-multilingual    | 512            |

如果你的文档经常超长，选一个输入限制大的模型，或者在 chunking 阶段把 chunk 做小。

## 主流模型横评

### 商业模型

**OpenAI text-embedding-3-large**

OpenAI 2024 年初的更新。效果确实强，MTEB 排行榜前几名的常客。

优点：

- 效果好，语义理解能力强
- 稳定性好，不用调参数
- 支持 8192 token 输入，长文档友好

缺点：

- 贵。$0.13 / 1M tokens（3-large），比上一代 ada-002 贵了 5 倍不止
- 需要 API 调用，延迟受网络影响
- 数据要出网，敏感场景慎用

**text-embedding-3-small**

3-large 的轻量版。效果比 3-large 差一截，但胜在便宜——$0.02 / 1M tokens。

适合什么场景：成本敏感的规模化应用、对效果要求不是特别高的 demo 或内部工具。

**Cohere embed-multilingual**

Cohere 的多语言模型。支持 100+ 语言，包括中文。

特点：

- 多语言效果比 OpenAI 好，尤其是小语种
- 输入限制 512 token，比较短
- 有 hume 这个面向企业的版本

适合什么场景：全球化产品、多语言知识库检索。

### 开源模型

**BGE（BAAI General Embedding）**

现在最流行的中文开源向量模型。系列包括：

- bge-base-zh：768 维，平衡型
- bge-large-zh：1024 维，效果更好但更慢
- bge-m3：支持 100+ 语言，融合了 dense + sparse + multi-vector

优点：

- 免费，能自己部署
- 中文效果很好（毕竟BAAI是中国的研究机构）
- 有量化版本，推理更快

缺点：

- 效果略逊于商业模型，尤其是长文本理解
- 需要自己运维 GPU 服务
- 稳定性取决于部署质量

**BGE-M3 的特殊之处**

这个模型把 dense（常规向量）、sparse（类似 BM25 的词权重）、multi-vector（一个文本输出多个向量）融合在一起了。实测在某些场景下召回效果确实更好，但推理也更慢。

我的建议是：先用 bge-base-zh 跑通业务，发现召回有瓶颈再考虑换 bge-large-zh 或 bge-m3。

**E5 和 E5-v2**

微软的向量模型。E5 ("Embeddings from bi-directional encoders") 特点是训练时就考虑了"查询-文档对"的匹配。

优势在于：专门为检索任务训练，in-context learning 能力强。

中文支持不如 BGE，国内用的人不多。

**MXBAI-embed-large**

2024 年后半程冒出来的黑马。效果接近 OpenAI 3-large，延迟更低，价格便宜。

适合什么场景：想用开源模型、但对效果要求比较高。

## 怎么选：几个决策框架

### 按效果选

如果效果是唯一追求，直接上 OpenAI 3-large。成本虽然高，但省去的运维和调试成本有时候更贵。

测试方式：用你的业务测试集跑 Recall@10，看哪个模型得分最高。

### 按成本选

大规模应用里，成本是个躲不过的问题。

一个简单的算术：假设你每天 100 万次 query，每次 query 检索 100 个 chunk。

- 用 OpenAI 3-large：100万 × 100 × $0.0001 ≈ $10,000/月（假设平均 chunk 100 tokens）
- 用 BGE-base 自己部署：GPU 成本约 $500-1000/月

差距是 10 倍。如果 query 量不大，省不了多少钱；如果量起来了，节省很可观。

### 按场景选

**场景1：短 query + 短 chunk**

这是最简单的场景，大部分模型都能 handle。选 BGE-base 或 3-small 够用了。

**场景2：长文档**

文档本身很长（比如 > 2000 tokens），query 相对短。

这时候要注意：

- 模型的最大输入长度能否覆盖你的 chunk size
- 模型在长文本上的语义理解能力

推荐：OpenAI 3-large（输入限制大）或 bge-m3。

**场景3：多语言**

要支持英文、中文、甚至小语种。

推荐：Cohere embed-multilingual 或 bge-m3。

**场景4：对延迟敏感**

线上服务对延迟要求高（比如 < 200ms）。

这时候有两个方向：

- 用小模型（3-small 或 bge-base）+ 量化
- 用本地部署的模型，避免网络开销

**场景5：数据敏感**

数据不能出网，必须本地部署。

只有一条路：开源模型 + 自己部署。BGE 是最稳的选择。

## 一个实用的测试方法

别信排行榜。信你自己的测试集。

测试流程：

1. **准备测试集**：从你的业务流量里抽 100-200 个有代表性的 query，每个 query 标注 3-5 个相关文档

2. **批量跑召回**：每个模型跑一遍，看 Recall@10 是多少

3. **看 bad case**：不仅看分数，还要看具体哪些 query 召回失败了，分析原因是"语义不对"还是"chunking问题"

4. **成本折算**：把延迟和成本算进去，看投入产出比

这个测试比任何排行榜都有说服力。

## 冷门但有用的细节

**句子向量 vs 段落向量**

同一个模型，对"一句话"和"一段话"的编码效果通常不同。短文本（一个句子）效果最好，长段落会因为"语义平均化"而效果下降。

如果你的 chunk 比较长（> 500 tokens），可以尝试：

- 把长 chunk 拆成多个短句，分别编码，取平均
- 用 bge-m3 这种专门优化过长文本的模型

**查询和文档用同一个模型还是不同模型？**

大多数情况下，用同一个模型。但有些模型会区分"查询编码器"和"文档编码器"（比如 GTR），训练时就针对这种场景优化。

对大多数场景，同一个模型足够。

**向量需要定期重训吗？**

看数据分布的变化。如果你的知识库内容变化不大，不需要重训。如果知识库经常大幅更新，建议定期（比如每季度）用新数据重训或微调模型。

## 小结

选向量模型这件事，看起来是选技术，实际上是权衡：

- 效果 vs 成本
- 托管 vs 自建
- 通用 vs 场景适配

没有绝对的最优解。但有一个绝对有效的办法：用自己的数据、自己的测试集、自己跑一遍评估。

很多团队的问题不是模型选错了，而是根本没测清楚。以为换个模型就能解决召回问题，结果换完发现——问题在 chunking 上，在 query 预处理上，就是不在向量模型上。

## 常见问题

**Q：Embedding 模型需要 Fine-tuning 吗？**

A：大多数场景不需要。通用模型在标准测试集上已经学到了足够的语义能力。但如果你的领域非常垂直（比如医疗、法律），用领域数据做轻量微调能提升明显。微调成本不高，拿 1000-2000 条标注数据就能看到效果。

**Q：同一批文档，需要定期重做向量吗？**

A：如果文档内容没变，就不需要。向量是一次性算好存着的。只有文档内容变了，或者换了模型，才需要重新算。

**Q：模型版本需要固定吗？**

A：强烈建议固定。向量模型的版本更新可能会改变输出的向量分布，导致同一段文本前后两次编码的结果不一致。如果你线上用了动态版本的模型，可能会出现"同样的 query、之前能召回的文档现在召回不了"的奇怪问题。

**Q：量化和向量化可以同时用吗？**

A：可以。很多向量数据库支持把向量存成 int8 或 float16 格式，精度损失很小。量化的好处是减少存储、加速计算，代价是 Recall 可能略有下降（通常 < 2%）。如果对召回率要求极高，用全精度；否则 int8 是性价比最高的选择。
